# issue-create: 访谈式 Issue 文档创建

通过结构化访谈帮用户把遇到的问题转化为高质量的 issue 文档。你是一个善于倾听的调查员——通过提问还原场景、补充细节、准确定性，最终产出一份完整的 issue 文档。

## 核心原则

1. **读代码辅助提问**：用户提到文件、模块、函数时，立即去读相关代码。用代码理解来提出更精准的问题，而不是问用户能自己查到的事
2. **不问显而易见的问题**：如果读代码就能知道的事（比如文件有多少行、函数签名是什么），不要问用户
3. **一次一个问题**：每个 AskUserQuestion 只问一件事，提供 2-4 个选项 + 推荐项
4. **所有面向用户的文本使用中文**
5. **访谈过程中可以穿插代码阅读**，不需要等访谈结束才读代码
6. **代码阅读是访谈的一部分**——读到有价值的信息时，用它来调整后续问题

## 访谈流程

### 阶段一：理解初始描述

用户通过 `$ARGUMENTS` 输入了他们遇到的问题。

1. **分析输入**：理解用户在描述什么类型的问题
2. **提取线索**：从描述中提取提到的文件名、模块名、函数名、错误信息等
3. **主动读代码**：对提到的文件/模块进行 Read/Grep，建立初步理解
4. **提出第一个问题**：基于代码理解，问用户最关键的一个缺失信息

**不要**：问"能不能详细说说？"这种开放式问题。用户已经给了初始描述，你应该通过代码阅读来理解上下文，然后问具体的、有技术深度的问题。

### 阶段二：场景还原

围绕问题类型选择最相关的维度。不要机械地逐项提问——根据已读代码的理解，只问真正缺失的信息。

**Bug/异常行为类**：
- 复现条件：必现还是偶发？是否有明确的触发步骤？
- 期望行为 vs 实际行为
- 错误信息、日志、堆栈（如果有）
- 影响范围：哪些用户/场景受影响？

**重构/技术债类**：
- 当前代码的主要问题是什么？（职责混乱？耦合过紧？重复代码？）
- 是否有具体的修改需求驱动（还是纯粹代码质量改进）？
- 修改时遇到过什么困难？

**性能类**：
- 在什么规模/负载下出现问题？
- 有没有具体的耗时数据或 profiling 结果？
- 是首次出现还是一直存在？

**安全类**：
- 具体的安全风险是什么？
- 是否有可被利用的攻击面？
- 影响的数据范围

### 阶段三：技术深化

1. **根据已收集的信息，主动读更多相关代码**
2. **提出有技术深度的问题**——这时候你已经读了代码，应该能问到点子上
3. **如果读代码时发现了用户没提到的问题**，可以补充询问
4. **尝试推断根因**，并向用户确认

**代码阅读建议**：
- 用户提到某个文件 → Read 全文
- 用户提到某个函数 → Grep 定义 + 读相关代码
- 用户提到某个错误消息 → Grep 搜索错误源头
- 不确定涉及哪些文件 → Glob 搜索 + Read 候选文件

### 阶段四：定性与评级

基于访谈和代码阅读，向用户确认 issue 的分类。如果用户没有强烈意见，agent 应基于代码阅读给出推荐。

**用 AskUserQuestion 确认以下维度**：

| 维度 | 选项 | 推荐逻辑 |
|------|------|----------|
| 类型 | Bug / 重构 / 技术债 / 性能 / 安全 / 文档 | 根据问题描述推荐 |
| 优先级 | 高 / 中 / 低 | 阻塞开发或影响用户→高；代码质量→中；优化→低 |

### 阶段五：生成文档

访谈完成后，按以下格式生成 issue 文档。

## Issue 文档格式

### 通用头部（所有类型）

```markdown
# [简洁描述问题的标题]

**状态**：Open
**优先级**：[高/中/低]
**创建日期**：YYYY-MM-DD

## 问题描述

[2-5 句话概括问题。包含：发生了什么、在什么条件下、影响是什么]

## 症状详情

[用表格、代码块、时间线等方式详细描述症状]
```

### Bug/异常行为类（追加）

```markdown
## 复现条件

- **复现频率**：[必现/偶发/仅在特定条件下]
- **触发步骤**：
  1. ...
  2. ...
- **环境**：[模型、OS、配置等]

## 根因分析

[如果通过代码阅读找到了根因，在这里描述。可以包含代码片段、事件流、对比分析等]

## 修复方案

[具体的修复建议，包含代码变更位置和改动内容]

## 相关代码

- `path/to/file.rs:line` —— [说明]
- `path/to/another.rs` —— [说明]
```

### 重构/技术债类（追加）

```markdown
## 现状数据

| 关注点 | 行数范围 | 行数 | 职责 |
|--------|---------|------|------|
| ... | ... | ... | ... |

[描述当前代码的问题：单文件过大？职责混乱？耦合过紧？]

## 拆分/改进方案

[具体的重构方案，可以包含模块结构图]

## 设计风险

[可能的风险和应对措施]

## 涉及文件

- `path/to/file.rs`（N 行）—— [说明]
- `path/to/another.rs` —— [说明]

## 收益

- **Locality**：[修改某功能时需要理解的范围]
- **Testability**：[可测试性改进]
- **认知负担**：[文件/模块行数变化]

## 验证标准

- [如何验证重构成功]
```

### 性能类（追加）

```markdown
## 性能数据

[具体的性能指标、测量数据、benchmark 结果]

## 瓶颈分析

[通过代码阅读分析的性能瓶颈]

## 优化方案

[具体的优化建议]

## 预期收益

[量化或定性的预期提升]
```

### 安全类（追加）

```markdown
## 风险描述

[具体的安全风险和攻击面]

## 影响范围

[受影响的数据、用户、系统]

## 修复方案

[安全修复建议]
```

## 文件命名规则

- 路径：`spec/issues/YYYY-MM-DD-<slug>.md`
- slug：从 issue 标题生成，使用 kebab-case，简洁描述问题
- 示例：`spec/issues/2026-05-11-streaming-text-invisible-with-tools.md`
- 如果 `spec/issues/` 目录不存在，先用 `folder_operations` 创建

## 完成后的输出

文档写入后，在终端输出简洁摘要：

```
✅ Issue 已创建 → spec/issues/YYYY-MM-DD-<slug>.md

类型：[Bug/重构/技术债/...] | 优先级：[高/中/低]
标题：[issue 标题]
涉及文件：N 个
```

## AskUserQuestion 格式规则

- 每个问题必须有推荐选项，放在第一位，标签后追加 `(推荐)`
- 描述字段以推荐理由开头
- 选项数量 2-4 个
- 示例：
  ```
  header: "Issue 类型"
  question: "这个问题属于什么类型？"
  options:
    - label: "Bug（推荐）"
      description: "这是一个可复现的异常行为，与预期不符"
    - label: "技术债"
      description: "代码质量问题，如职责混乱、重复代码等"
    - label: "重构"
      description: "结构性改进，如拆分大文件、解耦模块"
  ```

## 示例访谈流程

用户输入：`插件加载器的 load_enabled_plugins 函数太长了，1400 多行，改一个地方要读半天`

**Agent 内部处理**：
1. 读 `rust-agent-middlewares/src/plugin/loader.rs`
2. 确认确实 1418 行，函数职责混杂
3. 问用户：主要问题是什么？（职责混乱 / 文件过长 / 修改时影响面大）

用户选择：职责混乱

**Agent 内部处理**：
1. 继续读代码，识别主要职责
2. 问用户：是否需要拆分成多个模块？

...（继续访谈直到信息完整）

**最终输出**：`spec/issues/2026-05-11-plugin-loader-decomposition.md`

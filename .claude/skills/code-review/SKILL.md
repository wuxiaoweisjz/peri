# Code Review Skill

帮助 agent 完整执行一次代码审查，从分析到文档产出一气呵成。

---

## 第一步：读取项目上下文

在开始任何扫描之前，先快速获取项目背景，这样审查意见才有针对性：

**并行执行**：
```
Read("CLAUDE.md")          # 或 README.md / docs/ARCHITECTURE.md
Glob("**/Cargo.toml")     # 或 package.json / go.mod / pyproject.toml
bash("git log --oneline -20")   # 了解最近的改动方向（若是 git 仓库）
```

从这些信息中提取：
- 项目技术栈和架构模式
- 模块/Crate 之间的依赖关系
- 项目已知的约定和规范（命名风格、分层规则等）

---

## 第二步：确定 Review 模式

根据用户输入和项目上下文判断模式：

**探索型**：用户说"review 整个仓库"、"扫一下代码"、"找找问题"、没有指定具体文件
→ 跳到「探索型 Review 流程」

**定向型**：用户给出了具体文件、目录、PR diff、或"这段代码"
→ 跳到「定向型 Review 流程」

如果不确定，默认用探索型。

---

## 探索型 Review 流程

目标：先建立代码库全局认知，再深入问题集中区，避免盲目逐文件阅读。

### 阶段 1：结构侦察 + 模式扫描（全部并行执行）

**文件树分析：**
```
Glob("**/*")                  # 获取全量文件列表
Glob("**/*.{rs,ts,py,go}")    # 聚焦主要语言文件
```

从文件树中判断：
- 模块边界是否清晰，文件数量分布是否合理
- 命名风格是否一致（snake_case vs camelCase 混用）
- 某个目录是否过于臃肿（文件过多或单文件过大）

**常见问题 grep 扫描（并行执行）**：

```bash
# 错误处理缺失
Grep("unwrap\(\)|expect\(|panic!\(")
Grep("catch\s*\(\s*[a-z_]+\s*\)\s*\{\s*\}")   # 空 catch 块
Grep("TODO|FIXME|HACK|XXX")                   # 技术债标记

# 潜在性能问题
Grep("\.clone\(\).*\.clone\(\)")              # 热路径重复 clone
Grep("SELECT \*")                             # 全量数据库查询

# 代码气味
Grep("fn [a-zA-Z_]{40,}\(")                  # 函数名过长（暗示职责混乱）

# 安全问题
Grep("eval\(|exec\(|system\(")               # 命令注入风险
Grep("(password|secret|token|api_key)\s*=\s*[\"'][^\"']+[\"']")  # 硬编码凭证
```

### 阶段 2：选取热点文件深入阅读

根据扫描结果，选出 **3-8 个** 问题最集中的文件进行 `Read` 深入阅读。

**选取优先级**：
1. grep 命中次数多的文件
2. 单文件行数超过 500 行（上帝类/模块）
3. 核心业务逻辑文件（从文件名/结构推断）
4. 最近 git commit 中修改频率最高的文件

---

## 定向型 Review 流程

用户指定了目标，直接进入深度分析：

1. `Read` 读取目标文件（全量）
2. 若是 git diff，用 `bash("git diff HEAD~1")` 或用户提供的 diff
3. 若涉及多文件，先读入口文件，再追踪相关联的文件
4. 读完后按五大维度逐一审查（见下方）

---

## 五大 Review 维度

每个维度都有其核心关注点，**按严重程度排序**（高→低）。

### 1. 架构与设计

**关注点**：
- 职责是否清晰？一个模块/函数是否做了太多事
- 依赖方向是否合理？是否存在循环依赖或违反项目已有的分层规则
- 是否违反了项目既有的架构模式（如 CLAUDE.md 中描述的依赖关系）
- 抽象层次是否合适？是否过度抽象或抽象不足
- 接口设计：API 是否对调用者友好，错误信息是否可操作

**判断标准**：
- 为了测试一个函数需要 mock 5 个以上的依赖 → 耦合过紧
- 单文件超过 400 行 → 可能职责过多
- 改一个功能需要动 5 个以上的文件 → 可能有架构问题

### 2. 错误处理与健壮性

**关注点**：
- 错误是否被正确传播（不吞掉）
- panic/崩溃路径是否合理（生产代码中避免 unwrap）
- 边界条件：空值、零值、超大值、并发竞态
- 资源释放：文件句柄、连接、锁是否正确释放
- 错误信息是否有助于调试（包含上下文而非只有 "error"）

**Rust 特有**：
- `unwrap()`/`expect()` 在 lib 代码中应换成 `?` 传播
- `Arc<Mutex<>>` 死锁风险
- `async` 函数中 `.await` 后的错误处理

### 3. 性能

**关注点**：
- 热路径中的不必要分配（Vec/String clone、Box）
- O(n²) 或更差的算法（嵌套循环 + 集合操作）
- 同步阻塞（I/O、sleep、重型计算在 async 上下文中）
- 数据库/网络：N+1 查询、未分页的全量读取
- 缓存利用：重复计算相同结果

**判断标准**：只标注**热路径**中的性能问题，冷路径无需吹毛求疵。

### 4. 代码风格与可读性

**关注点**：
- 命名：变量/函数名是否自解释，缩写是否一致
- 函数长度：超过 50 行的函数应有充分理由
- 注释质量：注释解释"为什么"而非"是什么"；过时注释比没注释更危险
- 一致性：与同文件/同项目的风格是否一致
- 魔法数字/字符串：是否应提取为命名常量

**注意**：纯格式问题（缩进、空格）应由 linter/formatter 处理，不要在 review 中纠缠。

### 5. 脏代码与技术债

**关注点**：
- TODO/FIXME/HACK 注释：是否有 issue 跟踪，是否遗忘已久
- 死代码：注释掉的代码块、未使用的函数/变量
- 复制粘贴代码：相同逻辑出现 3 次以上（Rule of Three）
- 临时补丁：`if special_case`、`// 先这样吧` 类代码
- 过时依赖或废弃 API 的使用

---

## 严重程度标签

每个问题必须标注严重程度：

| 标签 | 含义 | 是否必须修复 |
|------|------|-------------|
| 🔴 **[critical]** | 可能导致 bug、安全漏洞、数据丢失 | 合并前必须修复 |
| 🟡 **[major]** | 明显影响可维护性或性能 | 强烈建议修复 |
| 🟢 **[minor]** | 代码气味、可读性问题 | 建议修复 |
| 💡 **[suggestion]** | 可选的改进方向 | 供参考 |

## 输出：写入 Review 文档

完成分析后，读取 `assets/review-template.md`（本 skill 目录下），按模板填充，然后：

1. 确定输出路径：默认写到项目根目录的 `reviews/YYYY-MM-DD-<topic>.md`
   - 若用户指定了路径，使用用户指定的路径
   - 若 `reviews/` 目录不存在，先用 `folder_operations` 创建

2. 用 `Write` 写入文件

3. 在终端输出**简洁摘要**：
   ```
   ✅ Review 完成 → reviews/2026-03-27-auth-module.md

   发现问题：3 critical / 5 major / 8 minor
   关键问题：
   - [critical] src/auth.rs:142 - unwrap() 可能导致线上 panic
   - [critical] src/db.rs:89 - SQL 查询未参数化
   - [major] src/handler.rs - 单文件 600 行，职责混乱
   ```

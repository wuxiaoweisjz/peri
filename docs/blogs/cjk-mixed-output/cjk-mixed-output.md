# 中转站模型为什么会输出日韩混杂的文字

你在用一个中转 API 写代码，返回的解释里突然夹了几个日文假名，或者一段韩文。模型没有换，参数没有动，但就是不说中文了。

这不是模型本身的问题，是中转站在获取模型访问权限时带进来的语言污染。

两条主要来源路径，污染机制各不相同。

## AI IDE 免费额度来源

Cursor、Windsurf 等 AI IDE 提供免费的模型访问额度。部分中转站通过批量注册账号、提取 IDE 内部的模型请求路径来获得可转售的 API 访问权。

问题出在账号注册地。IDE 服务商会根据账号注册时的 IP 或手机号归属地，在系统提示词里注入对应语言的指令——比如日本区账号的系统提示词是日文，韩国区账号是韩文。中转站批量收集账号时，日本、韩国的号比国内更容易注册，量也更大。这些账号的系统提示词就带着日文或韩文进来了。

中转站在转发请求时通常不会剥离或覆盖这些系统提示词。用户发的是中文请求，但模型收到的完整上下文里有一段日文系统提示词在最前面。模型会同时受到系统提示词语言和用户输入语言的影响，在两者之间漂移，输出里就出现了日韩文字混入中文的情况。

账号池越杂，轮转越随机，遇到异语言系统提示词的概率越高。用户每次请求打到哪个账号是随机的，所以同一个 prompt 有时输出纯中文，有时混日文，复现不稳定。

## 海外 Web 界面转发来源

另一类中转站用浏览器自动化（Playwright 之类的工具）访问 Claude.ai 或 ChatGPT 的 web 界面，用脚本把请求发进去、把回答抓出来再转发给用户。

这类方案要求挂代理访问海外服务。代理的出口 IP 在哪里，web 界面就呈现哪个地区的语言。出口在日本，Claude.ai 的界面就是日文，页面里注入给模型的上下文——包括功能说明、操作提示、UI 引导文字——全是日文。模型在这个日文界面环境里处理用户的中文请求，语言锚定就偏了。

更麻烦的是 session 复用。为了节省开销，很多这类中转站不会每次请求都开新的浏览器 session，而是复用已有的对话窗口。如果上一个用户的对话历史没有清干净，或者切换对话时 context 没有完全重置，前一段日韩文对话的残留会影响当前请求的输出语言。

## 根本原因

两条路径的污染源不同，但机制一样——**模型的语言倾向是由整个上下文决定的，不只是用户当前发的那条消息**。系统提示词是日文，或者 session 里有大量日文历史，模型就会往日文方向倾斜，哪怕用户问的是中文。

用直连官方 API 没有这个问题，因为系统提示词由自己控制，不会引入外来语言。中转站把控制权让给了来源，语言污染随着来源一起进来。
